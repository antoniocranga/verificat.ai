# CONSTITUTION.md — verificat.xyz (FactCheck.ro)

This document is the supreme law of this repository. Every other file — RULES.md, GEMINI.md, workflow docs, ADRs, even direct instructions given to an AI coding agent in a single session — is subordinate to this one. If anything conflicts with this Constitution, the Constitution wins. Changes to this file require explicit human sign-off; an AI agent must never edit this file as a side effect of another task.

## Article I — Product Truth

1. verificat.xyz is a **real-time claim verification assistant**, not a "truth detector." No feature, copy, UI element, or model prompt may imply the system delivers unilateral truth.
2. Every verdict shown to a user MUST be one of exactly six values: `True`, `Mostly True`, `Partially True`, `Misleading`, `False`, `Unverified`. No code path may emit any other verdict label, including for debugging UIs that ship to production.
3. Every verdict MUST carry, inseparably: a confidence score, at least one cited source, a human-readable evidence explanation. A verdict object missing any of these three fields is invalid and must not render.
4. `Unverified` is a first-class, expected outcome — not a failure state to be minimized at the cost of accuracy. A pipeline that never returns `Unverified` is broken, not good.
5. **No verdict may ever be generated by a single end-to-end LLM call.** Speech recognition, claim detection, evidence retrieval, and verdict generation are separate, independently testable, independently auditable pipeline stages (see ADR-006 in Notion). Any pull request that collapses this into a single opaque model call is rejected regardless of how good its outputs look in a demo.
6. Source trust scoring methodology must be documented and visible to the end user on request. Scores are not a black box.
7. All data must be anonymized in accordance with GDPR, including in local development and AI prompts.
8. **No Personally Identifiable Information (PII) is ever stored or processed unless absolutely necessary for the service functionality.**
9. **All data processed for AI training or inference must be anonymized and aggregated to remove any possibility of re-identification.**
10. **Explicit user consent is required for any data processing that may involve PII, even if anonymized.**
11. **Regular privacy audits must be conducted to ensure compliance with GDPR and best practices.**

## Article II — Security Non-Negotiables

1. No secret — API key, service-role key, database credential, OAuth client secret, signing key — is ever committed to git, pasted into Notion, pasted into chat with an AI assistant, or stored in plaintext anywhere outside the designated secrets manager. If this happens anyway, the secret is treated as compromised and rotated immediately; "but it was only seen by..." is not a defense.
2. Every database table has Row Level Security enabled with a **default-deny** posture. A table without RLS, or with an `RLS enabled, no policies` state (which Postgres treats as deny-all-but-owner, easy to misconfigure), must fail CI.
3. The Supabase `service_role` key (or equivalent) never reaches a client bundle — web, mobile, or extension. This is tested in CI, not just code-reviewed.
4. Authentication tokens are short-lived; refresh tokens rotate; revocation is real (checked server-side on every privileged request), not merely client-side.
5. Every Dockerfile runs its final stage as a non-root user. No exceptions for "it's just staging."
6. Dependency, container, and secret scanning are required CI checks on the default branch, not advisory ones.
7. Rate limiting is applied per-endpoint by default; an endpoint without a rate limit requires an explicit, reviewed exception comment explaining why.
8. CSP, HSTS, and secure-cookie flags are set at the reverse-proxy/middleware layer, not duplicated ad hoc per route.
9. **All data processing must be transparent, with clear documentation of data sources, processing steps, and retention policies.**
10. **Users must have the right to access, modify, and delete their data at any time.**
11. **Any third-party data sources must be disclosed, and users must be informed if their data is being compared against external sources.**
12. **No data may be shared with third parties without explicit user consent.**

## Article III — Engineering Discipline

1. **No disconnected work.** Every task in the Notion Engineering Backlog must declare its dependencies, and nothing may be marked "In Progress" while a P0 dependency is "Not started."
2. Every Task/Subtask requires, before it can leave "Not Ready": a Description, Acceptance Criteria, Technical Notes, Dependencies, Security Considerations, and Testing Requirements. An AI agent picking up backlog work must refuse to start a task missing any of these fields and should flag it for human completion instead of guessing.
3. Technology choices are recorded as ADRs (Architecture Decision Records) in Notion before implementation, not reverse-engineered as documentation after the fact.
4. Romanian-language correctness (diacritics: ă â î ș ț; Daco-Romanian standard) is a first-class quality bar across UI copy, STT post-processing, and claim normalization — not an afterthought localization pass.
5. Accessibility (WCAG 2.1 AA as a floor) is a launch requirement for the public website, not a nice-to-have.

## Article IV — AI Agent Conduct (Antigravity / Gemini and any other coding agent)

1. An agent operating in this repo follows GEMINI.md and the workflow docs in `/workflows` for *how* to work, and this Constitution for *what is and isn't negotiable* while doing it.
2. An agent must never weaken a security control (disable RLS, widen a CSP, remove a rate limit, add a secret to a committed file) to make a task "pass" faster. If a security control blocks a task, the correct move is to flag the conflict to a human, not route around it.
3. An agent must not mark a Notion backlog item "Done" without the corresponding Acceptance Criteria and Testing Requirements actually being satisfied — not merely "code written."
4. An agent must treat any credential it encounters in context (chat, files, logs) the same way a human engineer would: never re-emit it in full, never write it to a new file or commit, and flag it as needing rotation if it appears to have been exposed outside the secrets manager.

## Amendment Process

Amendments require a written rationale, a human reviewer, and a corresponding update to any RULES.md or workflow doc that the change affects. An AI agent may *propose* an amendment as a suggestion; it may not apply one.
